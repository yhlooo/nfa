# 自动文档更新

你的任务是系统性地审查并更新指定的文档，确保其与当前代码库完全一致。你必须严格区分“用户关心的功能”和“开发者关心的实现细节”。

## 核心工作流程

### 第1步：解析文档
1. 读取 $ARGUMENTS 指定的目标文档。如果未提供参数，扫描项目根目录及常见子目录（如 `docs/`, `documentation/`, `guides/`），引导用户选择要更新的文档。
2. 提取文档大纲，识别所有描述“功能点”的段落、列表项或小节。
3. 将每个功能点表述为一条可验证的断言（例如：“支持三种排序方式：日期、名称、大小”）。

### 第2步：代码验证
对每个功能点，在代码库中执行多维度搜索：
- **关键字符串**：搜索功能名称、描述中的关键词、错误消息或日志文本。
- **配置文件**：检查 `config`、`settings`、`constants` 及相关模型定义文件。
- **接口定义**：搜索 API 路由、CLI 参数解析库（如 argparse、click、cobra）、GraphQL schema 等。
- **测试文件**：快速浏览相关测试用例，它们往往精确描述了预期行为。

**验证结果分类处理**：
- **✅ 完全匹配**：代码实现与文档描述一致。无需操作。
- **⚠️ 部分过时**：参数名、默认值或行为逻辑已变更。记录需修改的内容。
- **❌ 功能已移除**：在代码中找不到任何对应实现。确认非重命名或迁移后，标记为“需从文档删除”。
- **🆕 发现新功能**：代码中存在文档未提及的用户可见功能。

### 第3步：发掘遗漏功能
使用以下策略主动搜索文档未覆盖的用户可见功能：
1. **分析入口点**：
    - CLI 工具：解析主命令的参数列表和子命令。
    - Web API：提取所有已注册的路由或中间件栈。
    - 库/SDK：检查模块的公开导出、类的公有方法。
2. **扫描变更日志**：读取 `CHANGELOG.md` 或 `CHANGES.md`，找出近期版本新增特性。
3. **检查配置文件模板**：分析 `*.example`、`*.template` 或 `*.dist` 文件，找出可配置的选项。
4. **对比环境变量文档**：搜索 `process.env`、`os.environ` 等，找出可配置的环境变量。

对每个候选功能点，快速验证其确实影响最终用户或运维人员，而非纯内部辅助函数。

### 第4步：更新文档
执行修改时，严格遵守 `principles.md` 中的编写原则。关键要求：
- **必须面向用户**：直接说明能做什么、如何做。补充简短的示例代码块。
- **必须过滤细节**：删除或概略化内部实现、类结构、非公开方法的描述。
- **必须保持结构**：更新后文档结构清晰，层级合理，不引入杂乱内容。

**修改策略**：
- 对离散的修正（如改参数名），直接使用 Edit 工具原位修改。
- 对大面积过时或需补充整个小节，可使用 Write 重写完整章节后替换。
- 优先保留原文档的写作风格和语气。

### 第5步：总结报告
完成更新后，生成一个简短的更新报告，格式如下：

```markdown
## 📋 文档更新报告

**文档**：[文档路径]

### 修正（代码与文档不一致）
1. [修正前描述 → 修正后描述]
2. ...

### 新增（文档中未提及的功能）
1. [功能简述]
2. ...

### 移除（代码中已不存在的功能）
1. [功能简述]
2. ...

### 待确认
1. [无法完全确定、需要人工复查的点]
```

最后，提示用户使用 `git diff` 审查更改。
