# 文档编写原则

在更新文档时，必须遵循以下原则，确保产出的文档对最终用户有价值。

## 面向用户，而非开发者

### ✅ 应该包含的内容
- **做什么 (What)**：功能解决了什么问题，在什么场景下使用。
- **怎么用 (How)**：清晰的步骤、命令行示例、配置文件片段、API 请求/响应示例。
- **入参与出参**：所有用户可控制的参数、选项、环境变量及其默认值、可选值。
- **限制与边界**：使用上限、兼容性要求、已知行为差异。
- **前置条件**：权限、依赖、版本要求。

### ❌ 应该忽略或简化的内容
- **内部实现**：算法选择、设计模式、调用链、文件内部结构。
- **代码重构细节**：仅影响开发者阅读代码的变更，如拆分函数、重命名变量。
- **非公开接口**：私有方法、内部 API 端点、仅供其他内部模块调用的函数。
- **过度技术细节**：数据库 SQL 语句、内存管理机制、编译优化参数。

**示例对比**：
> ❌ 实现细节版：`UserService` 类调用 `PasswordHasher` 的 `bcrypt_verify` 方法，通过 `DatabaseConnectionPool` 获取连接执行 `SELECT * FROM users WHERE email = ?` 查询。
>
> ✅ 用户导向版：系统验证用户输入的邮箱和密码。密码采用行业标准加密存储，登录过程自动且安全。

## 结构清晰，易于扫描
- 使用描述性的标题和小标题。
- 列表项保持简短，一个列表项描述一个要点。
- 关键术语在首次出现时使用**粗体**。
- 代码块必须指定语言，确保语法高亮。
- 命令行示例应可直接复制粘贴执行。

## 准确性与完整性
- 参数、方法名、返回值必须与当前代码完全一致，区分大小写。
- 默认值必须准确。
- 若功能存在多种使用方式（如 SDK 和 CLI），优先展示最常用的一种，并在合适位置提及其他方式。
- 错误码或常见错误场景若有明确提示，应一并记录。

## 维护友好
- 如涉及多个文档页面，确保更新后交叉链接仍然有效。
- 避免在文档中直接嵌入可能频繁变动的代码行数或文件路径。
- 对于大型功能，优先提供“快速开始”示例，再提供“完整选项”参考。
