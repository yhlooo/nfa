## Context

当前项目的所有文档都集中在 README.md 中，包括项目介绍、配置说明和自定义技能的使用指南。随着项目发展，README.md 将变得冗长且难以维护。项目需要一个结构化的文档系统来提升文档的可维护性和可读性。

当前状态：
- README.md 包含：项目简介、配置说明、自定义技能完整使用指南
- 无独立 docs/ 目录

## Goals / Non-Goals

**Goals:**
- 建立清晰的文档目录结构（tutorials/、guides/、reference/）
- 迁移现有文档到合适的位置
- 保持 README.md 简洁，仅包含项目概览和快速链接
- 确保文档迁移后用户仍能方便地找到所需信息

**Non-Goals:**
- 暂不创建新的教程内容（仅建立目录结构）
- 不涉及代码变更
- 不改变现有文档的技术内容（仅迁移和重新组织）

## Decisions

### 1. 文档迁移策略

**决策**: 将 README.md 中的自定义技能说明迁移到 `docs/guides/skills.md`，配置说明保留在 README.md（作为快速开始的一部分）

**理由**:
- 自定义技能是独立的功能特性，适合放在 guides/ 中
- 配置说明是快速使用的基础，保留在 README.md 便于用户快速上手
- 配置文件的详细结构定义放在 `docs/reference/` 中

**考虑的替代方案**: 将配置说明也移到 guides/
- 缺点：用户需要多次跳转才能开始使用项目
- 决定：配置基础说明保留在 README，详细配置说明放到 reference/

### 2. 目录命名约定

**决策**: 使用小写字母和连字符命名目录和文件（如 `skills.md` 而非 `Skills.md`）

**理由**:
- 与项目现有的 kebab-case 命名风格保持一致
- 便于在不同操作系统间兼容（避免大小写敏感问题）

### 3. 教程文件命名

**决策**: 教程文件使用数字前缀命名（如 `01-getting-started.md`）

**理由**:
- 数字前缀确保文件系统排序与阅读顺序一致
- 用户可以按数字顺序依次阅读

### 4. README.md 重新设计

**决策**: README.md 将包含以下部分：
1. 项目简介
2. 快速开始（包含基础配置和运行命令）
3. 文档导航（指向 docs/ 目录各子目录）
4. 项目状态（当前处于早期阶段）

**理由**:
- 保持简洁，提供最关键的信息
- 引导用户到 docs/ 获取详细内容

## Risks / Trade-offs

### 风险 1: 旧链接失效
**风险**: 如果用户或外部资源有指向 README.md 中特定章节的链接，迁移后这些链接会失效

**缓解措施**:
- 当前项目处于早期阶段，外部链接较少
- 在 README.md 中添加迁移说明，告知用户新的文档位置

### 风险 2: 用户需要适应新结构
**风险**: 用户需要时间适应新的文档结构

**缓解措施**:
- 在 README.md 中提供清晰的文档导航
- 使用直观的目录名称和文件名
- 确保 README.md 中包含足够的快速开始信息

## Migration Plan

### 迁移步骤

1. 创建目录结构
   ```
   docs/
   ├── tutorials/
   ├── guides/
   │   └── skills.md
   └── reference/
       └── config.md
   ```

2. 创建 `docs/guides/skills.md`
   - 将 README.md 中 "自定义技能 (Custom Skills)" 章节的内容迁移至此

3. 创建 `docs/reference/config.md`
   - 提取配置结构说明，详细描述 `~/.nfa/nfa.json` 的配置项

4. 修改 README.md
   - 保留项目简介
   - 简化快速开始部分
   - 移除自定义技能详细说明，保留简短提及和链接
   - 添加文档导航章节

5. 验证
   - 确保所有文档链接正确
   - 确保从 README.md 可以导航到所有文档

### 回滚策略
如需回滚，只需：
1. 将 `docs/guides/skills.md` 内容恢复到 README.md
2. 删除 `docs/` 目录
3. 恢复 README.md 到原始状态
