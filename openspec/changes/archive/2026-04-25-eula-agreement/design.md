## Context

NFA 当前没有用户协议确认机制，启动后直接进入 TUI 或执行用户指令。作为面向终端用户的金融 AI 工具，需要在首次使用前获得用户对使用条款的明确同意。

当前 `PersistentPreRunE`（`pkg/commands/root.go:105-155`）已包含启动前初始化序列：创建数据目录、初始化日志、加载配置、设置 i18n。EULA 检查适合插入在 i18n 之后，因为提示信息需要国际化支持。

项目已有 3 处 embed 使用先例（`pkg/otter/`、`pkg/i18n/locales.go`、`pkg/skills/skill_loader.go`），EULA 模块可以遵循相同模式。

## Goals / Non-Goals

**Goals:**
- 首次启动时向用户展示协议内容并要求明确同意
- 用户同意后记录协议版本，后续相同版本自动跳过
- 协议更新后检测变更并要求用户重新确认
- 每次启动将当前协议写入 `~/.nfa/eula.md` 便于查阅
- 仅对 agent 主命令生效，管理子命令（`version`、`models list` 等）不受影响

**Non-Goals:**
- 不提供"稍后同意"或部分同意选项（仅二元选择）
- 不在 TUI 内部显示协议（使用简单终端 I/O）
- 不提供 GUI 同意界面
- 不支持多用户或权限分离
- 不修改现有子命令的行为

## Decisions

### 1. EULA 检查插入位置：`PersistentPreRunE` 中，i18n 初始化之后

**选择**：在 `PersistentPreRunE` 中调用 `eula.Check()`，通过 `cmd.Name()` 判断是否为主命令。

**原因**：
- i18n 必须就绪才能显示多语言提示
- 数据目录（`~/.nfa/`）必须存在才能写入 `eula.md` 和签名文件
- 日志必须初始化才能记录 EULA 相关日志
- `PersistentPreRunE` 在 `RunE` 之前执行，还没启动 TUI，使用简单终端 I/O 即可

**替代方案**：在根命令的 `RunE` 中检查。不采用的原因是希望尽早拦截，避免不必要的初始化。

### 2. 交互方式：`fmt.Print` + `bufio.Scanner`

**选择**：使用标准库的终端 I/O 而非集成到 bubbletea TUI 中。

**原因**：
- EULA 检查时 TUI 尚未启动，无法使用 bubbletea
- 简单 I/O 对 `-p`（非交互模式）同样适用
- 减少对 TUI 组件的侵入性修改
- 用户同意/拒绝是简单的 y/n 选择，不需要复杂 UI

### 3. 版本跟踪：SHA256 签名文件

**选择**：计算 `eula.md` 内嵌内容的 SHA256，将哈希值写入 `~/.nfa/eula_signed.sha256sum`。

**原因**：
- SHA256 对内容变更敏感，任何文本修改都会导致不匹配
- 仅存储哈希值，不需要维护版本号
- 文件名自描述：`.sha256sum` 后缀明确表示内容

**替代方案**：
- 版本号：需要手动维护版本号，容易遗漏更新导致不一致，不采用
- 文件修改时间：不可靠，不同部署时间不同，不采用
- 完整协议文本副本对比：占用空间大，SHA256 更简洁

### 4. `eula.md` 写入策略：每次启动覆盖写入

**选择**：每次启动都将内嵌的 `eula.md` 写入 `~/.nfa/eula.md`。

**原因**：
- 确保用户总能查看最新版本
- 写入操作开销极小（几 KB）
- 简单可靠，不需要额外的版本判断逻辑

### 5. 模块结构：独立的 `pkg/eula` 包

**选择**：新建 `pkg/eula/` 包，包含 `eula.go`（逻辑）、`eula.md`（协议内容）、`i18n.go`（提示消息）。

**原因**：
- 职责单一，与现有模块无耦合
- 遵循项目现有的 embed + i18n 模式
- 方便单独测试

## Risks / Trade-offs

- **风险**：用户在非交互环境（如 CI/CD）中运行 `nfa -p "问题"` 时，EULA 提示会阻塞等待输入 → 可通过标准输入重定向检测或增加 `--accept-eula` 参数缓解，但当前版本不需要支持 CI 场景
- **风险**：用户手动删除 `~/.nfa/eula_signed.sha256sum` 后会重新看到协议提示 → 这是预期行为，不是问题
- **权衡**：管理子命令（`version`、`models list`）不检查 EULA，意味着用户可以先运行 `nfa version` 再通过某种方式绕过。但 EULA 的目的不是强制技术壁垒，而是诚实的使用告知，这种程度的"绕过"是可接受的
